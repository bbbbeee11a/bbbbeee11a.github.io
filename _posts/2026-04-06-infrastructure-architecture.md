## 인프라 vs 인프라 아키텍처

IT 인프라란 IT의 기반이 되는 것 전반을 말한다. 그런데 공부하면서 인프라와 인프라 아키텍처가 다른 개념이라는 걸 처음으로 명확하게 인식했다.

인프라 아키텍처는 IT 인프라의 **구조**, 즉 실제 구현 방식이다. 서로 다른 시스템들이 거의 같은 구조를 가지고 동작한다는 점에서, 아키텍처는 인프라를 어떻게 설계하고 배치하느냐의 문제다.

---

## 집약형 아키텍처

하나의 대형 컴퓨터(메인프레임)로 모든 처리를 담당하는 방식이다.

- 장비 고장으로 업무가 중단되지 않도록 주요 부품을 **다중화**해야 한다
- 하나의 처리가 대량의 요청을 보내더라도 다른 처리에 영향을 주지 않도록 **유한 리소스 관리**가 필요하다

| 장점 | 단점 |
|---|---|
| 구성이 단순하고 안정성·성능이 높음 | 비용이 많이 들고 확장성에 한계가 있음 |

---

## 분할형 아키텍처

여러 대의 컴퓨터를 조합해서 하나의 시스템을 구축하는 방식이다. 표준 OS와 개발 언어를 사용하기 때문에 **오픈 시스템**, 여러 컴퓨터를 연결해서 사용하기 때문에 **분산 시스템**이라고도 부른다.

| 장점 | 단점 |
|---|---|
| 저렴한 장비로 비용 절감 가능, 확장성이 좋음 | 서버 수가 늘수록 운영 구조가 복잡해짐 |

서버를 분할하는 방식은 크게 **수직형**과 **수평형**으로 나뉜다.

> **물리 서버 vs 논리 서버**
> 서버는 특정 역할에 특화된 것(웹 서버, DB 서버 등)을 의미하고, 컴퓨터 자체는 물리 서버라고 부른다. 일반적인 IA 서버(Intel x86)는 일반 PC와 아키텍처가 동일하다.

---

## 수직 분할형 아키텍처

역할에 따라 **계층(위아래)으로 분리**하는 방식이다.

### 클라이언트-서버형 (C/S)

클라이언트가 물리 서버에서 운영 중인 애플리케이션이나 데이터베이스에 접속해서 이용하는 형태다.

- 클라이언트 측에 전용 소프트웨어를 설치해야 한다
- 클라이언트에서 처리를 하고 서버에서 데이터를 취득하므로 서버 부하가 낮다
- 애플리케이션 갱신 시 클라이언트 소프트웨어 업데이트가 필요하다
- 서버에 처리가 집중되면 확장성에 한계가 생긴다

### 3계층형 아키텍처

C/S형의 단점을 개선한 구조로, 세 가지 계층으로 역할을 분리한다.

| 계층 | 역할 |
|---|---|
| 프레젠테이션 계층 | 사용자 입력 수신, 화면 표시 |
| 애플리케이션 계층 | 사용자 요청에 따른 업무 처리 |
| 데이터 계층 | 데이터 입출력 담당 |

특정 서버에 부하가 집중되는 문제가 해결되고, 클라이언트 업데이트가 필요 없어진다. 이미지 파일만 읽으면 되는 경우처럼 모든 요청이 AP 서버나 DB 서버를 거치지 않아도 된다는 점도 장점이다. 다만 전체 구조가 C/S형보다 복잡해진다.

---

## 수평 분할형 아키텍처

용도가 같은 서버를 늘려나가는 방식으로, 시스템 안정성과 전체 성능을 높인다. 수직 분할형과 배타적인 관계가 아니라 **두 방식을 함께 채택**하는 것이 일반적이다.

### 단순 수평 분할형 (샤딩 / 파티셔닝)

같은 기능을 가진 독립적인 시스템을 여러 개 단순 분할하는 방식이다.

- 한쪽 시스템에 장애가 발생해도 다른 쪽에 영향이 없어 **독립성이 높다**
- 같은 애플리케이션을 운영 중이라면 양쪽 시스템 모두 업데이트해야 한다
- 데이터를 독립적으로 보유하며, 지리적으로 멀리 떨어진 시스템에 주로 활용한다
- 한 시스템에만 부하가 몰리면 처리 성능이 2배가 됐다고 볼 수 없다

### 공유형 아키텍처

단순 분할형과 달리 일부 계층에서 상호 접속이 이루어진다. 데이터 계층을 공유할 경우 양쪽에서 데이터를 참조하고 동기 처리가 필요하다.

- 시스템 간 독립성이 낮아진다
- 공유한 계층의 확장성이 낮아진다는 단점이 있다

---

## 지리 분할형 아키텍처

업무 연속성과 시스템 가용성을 높이기 위해 **지리적으로 분할**하는 아키텍처다.

### 스탠바이형 (HA 구성 / 액티브-스탠바이)

물리 서버를 최소 두 대 준비하여 한 대가 고장 나면 소프트웨어를 다른 서버로 옮겨 운영하는 방식이다. 이때 소프트웨어 재시작을 자동으로 처리하는 구조를 **Failover**라고 한다.

스탠바이 서버가 평소에 놀고 있어 리소스 낭비가 발생할 수 있다. 이를 개선하기 위해 양쪽을 교차로 사용하다가 한쪽이 고장 나면 나머지가 전체를 처리하는 방식을 택하기도 한다.

### 재해 대책형 아키텍처

특정 데이터 센터에 재해가 발생했을 때 다른 사이트의 재해 대책 환경에서 업무를 재개하는 방식이다.

- 별도 사이트에 서버를 최소 구성 혹은 동일 구성으로 배치한다
- 애플리케이션과 데이터 최신화가 필요하며, 특히 데이터는 사이트 간 **실시간 동기 처리**가 중요하다

---

## 마치며

인프라를 안다고 생각했는데, 막상 아키텍처 유형을 정리하다 보니 내가 알고 있던 건 단편적인 지식이었다는 걸 느꼈다. 특히 스케일 업/아웃이 수직/수평 분할과 같은 개념인 줄 알았는데 엄연히 다르다. 스케일 업/아웃은 리소스 확장 방식이고, 수직/수평 분할은 시스템 구조를 어떻게 나누느냐의 문제다.

오늘 공부를 통해 인프라를 구성하는 방식과 그 구체적인 유형들을 처음으로 체계적으로 이해한 것 같다. 각 아키텍처가 단독으로 존재하는 게 아니라 실제 시스템에서는 여러 방식을 조합해서 사용한다는 점도 인상적이었다.
