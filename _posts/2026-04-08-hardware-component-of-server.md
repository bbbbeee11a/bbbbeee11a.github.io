## 물리 서버

### 서버 외관과 설치 장소

서버는 랙(Rack)에 장착된다. 랙에는 서버 외에도 HDD가 모여 있는 스토리지(Storage), 인터넷 및 LAN을 연결하기 위한 네트워크 스위치 등도 탑재되어 있다.

### 서버 내부 구성

서버 내부는 CPU, PCI, HDD, FAN, 메모리, 전원, NIC 등으로 구성되어 있으며, 이 컴포넌트들을 연결하는 회선을 **버스(Bus)**라고 한다.

- CPU와 메모리는 물리적으로 직접 연결되어 있다
- PCI Express 슬롯은 외부 장치를 연결하는 인터페이스로, Intel Xeon 아키텍처에서는 CPU가 PCI를 직접 제어한다
- **BMC(Baseboard Management Controller)**는 서버 하드웨어 상태를 독립적으로 감시한다. 장애 발생 시 BMC 콘솔이나 네트워크를 통해 원격으로 서버 상태를 확인하거나 재시작할 수 있다

서버와 PC는 물리적인 기본 구성이 같다. 차이점은 전원이 이중화되어 있고, 대용량 CPU와 메모리가 탑재되어 있다는 점이다.

> **펌웨어 vs 드라이버**
> - **펌웨어(Firmware)**: 하드웨어 장치 내부 메모리에 저장되는 기본 제어 소프트웨어
> - **드라이버(Driver)**: OS가 하드웨어를 사용할 수 있도록 OS 내부에 설치되는 소프트웨어

## CPU

CPU(Central Processing Unit)는 서버의 중심에 위치하여 명령을 받아 연산을 실행하고 결과를 반환한다. 명령과 데이터는 기억 장치나 입출력 장치를 통해 전달된다.

현재는 CPU를 코어(Core)라 부르며, 각 코어는 독립적인 처리를 할 수 있다.

명령 체계는 다음과 같다:

사용자 입력 / 프로세스(웹 서버, DB 등)
↓
OS (Operating System)
↓
CPU

인터럽트(Interrupt)가 발생하면 CPU는 현재 작업을 중단하고 해당 인터럽트를 처리한다.

## 메모리

메모리는 CPU 옆에 위치하며, CPU에 전달할 명령이나 데이터를 저장하고 처리 결과를 받는다.

메모리에 저장된 정보는 **휘발성**이라 서버를 재시작하면 사라진다. 그럼에도 메모리를 사용하는 이유는 다음과 같다:

- 액세스 속도가 매우 빠르다
- 물리적인 모터 구동 없이 전기적인 처리만으로 데이터를 저장한다

### 캐시 계층 구조

CPU는 내부적으로 레지스터와 L1/L2 캐시라는 메모리를 자체적으로 가지고 있다. 메모리보다 빠르지만 용량이 작고 비싸다. 용량, 가격, 속도는 트레이드오프 관계다.

메모리에 액세스하려면 메모리 컨트롤러를 통해 CPU 밖으로 나가야 한다. 이 과정에서 발생하는 레이턴시를 줄이기 위해 자주 사용하는 명령과 데이터를 코어에 가까운 곳에 배치한다.

| 캐시 | 특징 | 범위 |
|---|---|---|
| L1 | 초고속 | 각 코어 전용 |
| L2 | 고속 | 각 코어 전용 |
| L3 | 준고속 | CPU 전체 공유 |

캐시를 여러 단계로 나눈 이유는 캐시가 커질수록 액세스 속도가 느려지기 때문이다. 초고속으로 접근할 데이터는 L1에, 준고속으로 접근할 데이터는 L2에 두는 방식으로 효율을 높인다.

### 메모리 인터리빙 (Memory Interleaving)

여러 채널과 뱅크를 병렬로 활용해서 데이터 전송 대역폭과 처리 속도를 향상시키는 기술이다.

[프로그램]
Data1, Data2, Data3, Data4 순서로 접근
↓
CPU: "연속 접근이네?" → prefetch 가능
↓
메모리: A, B 채널에 데이터가 나뉘어 저장
↓
병렬로 더 빠르게 CPU에 공급

## I/O 장치

### HDD (Hard Disk Drive)

메모리에 비해 CPU에서 멀리 배치되며, 장기 저장 목적으로 사용된다.

- 메모리: 빠르지만 휘발성 (전원이 없으면 데이터 소멸)
- HDD: 느리지만 비휘발성 (전원이 없어도 데이터 유지)

HDD는 자기 원판이 고속으로 회전하며 읽기/쓰기를 처리하기 때문에 속도가 물리 법칙에 의해 좌우된다. SSD는 메모리처럼 반도체로 만들어졌지만 비휘발성이라는 HDD의 장점도 가진다.

HDD가 많이 탑재된 하드웨어를 **스토리지(Storage)**라고 부르며, I/O 서브시스템이라고도 한다. 내부에 CPU와 캐시가 존재하고, 서버와 I/O 시에는 HDD가 직접 데이터를 교환하는 게 아니라 캐시를 통해 처리한다.

대형 스토리지와 연결할 때는 **파이버 채널(Fiber Channel, FC)** 케이블을 사용해 **SAN(Storage Area Network)**이라는 네트워크를 경유한다. 서버에 FC 포트가 없을 경우 PCI 슬롯에 **HBA** 카드를 삽입해 접속한다.

#### I/O 캐시 방식

**라이트 백 (Write Back)**
- 쓰기 시 캐시에만 기록하고 완료로 간주
- 장점: 고속 I/O 실현
- 단점: 캐시만 기록된 상태에서 장애 발생 시 데이터 손실 가능 → 별도 캐시 미러링으로 안정성 확보

**라이트 쓰루 (Write Through)**
- 쓰기 시 캐시와 HDD 모두에 기록
- 장점: 데이터 안전성이 높음
- 단점: 라이트 백 대비 쓰기 속도가 느림

### 네트워크 인터페이스

서버와 외부 장비(다른 서버, 스토리지 등)를 연결하기 위한 외부 접속용 인터페이스다.

### I/O 제어

Intel Xeon 아키텍처에는 **PCH(Platform Controller Hub)**라는 칩셋이 탑재되어 있어, CPU가 담당하는 메모리와 PCIe 외의 I/O 제어를 처리한다. CPU 외에 별도 컨트롤러가 존재하는 이유는 CPU가 연산에 집중할 수 있도록 역할을 분담하기 위해서다.

> **PCI x8, x16의 의미**
> PCI 슬롯에 연결되는 I/O 회선(레인)의 수를 의미한다. 숫자가 클수록 더 많은 데이터를 동시에 전송할 수 있다.

## 버스 (Bus)

버스는 서버 내부 컴포넌트들을 연결하는 회선이다. 버스에서 가장 중요한 것은 **데이터 전송 능력(대역)**이다.

대역(Throughput) = 전송폭 × 전송 횟수 

전송 횟수 = 1초 / 1처리당 소요 시간(응답시간)

CPU에 가까울수록 버스 대역이 크고, 버스 흐름에서 장치 사이에 병목 현상이 없어야 한다.

## 마치며

오늘 공부는 서버 내부 구조를 처음으로 체계적으로 들여다본 시간이었다. 평소에 CPU, 메모리, HDD라는 단어를 아무렇지 않게 써왔는데, 실제로 이 컴포넌트들이 어떻게 연결되고 데이터가 어떤 경로로 흐르는지는 생각해본 적이 없었다.

특히 캐시 계층 구조가 인상적이었다. L1/L2/L3로 나눈 이유가 단순히 "빠른 것은 비싸니까"가 아니라 레이턴시를 최소화하기 위한 설계라는 걸 이해하니 훨씬 자연스럽게 받아들여졌다. HDD에 있는 데이터가 CPU에 도달하기까지 버스, 캐시, 메모리 컨트롤러를 거치는 과정을 생각하면 고성능 시스템이 단순히 스펙 숫자가 아니라 이 전체 경로의 병목을 줄이는 싸움이라는 느낌이 들었다.
